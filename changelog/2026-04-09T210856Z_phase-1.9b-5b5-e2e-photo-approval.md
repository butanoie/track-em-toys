## Changes Implemented

### 1. New E2E spec: `web/e2e/curator-photo-approvals.spec.ts`

Eight scenarios grouped into four `test.describe` blocks:

| # | Scenario | Describe group |
|---|----------|----------------|
| S1 | Approve public via A hotkey → photo removed from queue | Photo approval decisions |
| S2 | Approve training-only via T hotkey → PATCH body has `visibility: 'training_only'` | Photo approval decisions |
| S3 | Reject via R → click Blurry → PATCH body has `rejection_reason_code: 'blurry'`, no text | Photo approval decisions |
| S4 | `intent: 'training_only'` disables public Approve button with training-only tooltip | Decision gating |
| S5 | `existing_photos[0].distance = 3` shows near-duplicate banner (threshold 4) | Warnings and empty states |
| S6 | Approving the only pending photo renders the empty state | Warnings and empty states |
| S7 | `can_decide: false` disables all 3 decision buttons, keeps Prev/Next enabled | Decision gating |
| S8 | Primed 409 response shows conflict banner, Dismiss button clears it | Conflict handling |

Design decisions:

- **Curator project, not admin** — semantic match with the real
  `requireRole('curator')` guard, auto-matched by the existing
  `curator-*.spec.ts` regex in `playwright.config.ts` (no config change).
- **Fresh `MockPhotoApprovalState` per test** — hermetic, each test
  can run under `test.only`, safe under `fullyParallel: true`.
- **Factory helpers inline in spec** (`makePendingPhoto`,
  `makeTrainingOnlyPhoto`, `makeNearDuplicatePhoto`,
  `makeSelfReviewPhoto`) — only one spec consumes them today;
  promote to `mock-helpers.ts` if a second caller ever appears.
- **`page.addInitScript` seeds `localStorage['photo-approval-shortcuts-seen'] = 'true'`
  before `goto`** so the first-visit keyboard overlay dialog doesn't
  intercept decision hotkeys.
- **Locator discipline**: `role="alert"` is used for both the
  near-duplicate banner and the 409 conflict banner — every assertion
  filters by text (`/Possible duplicate/` vs `/no longer pending/`),
  documented in the spec header comment.
- **`Promise.all([waitForRequest, keyboard.press])` pattern** for
  decision tests — attaches the request waiter before firing the key
  so the race between key handler and PATCH dispatch is closed. All
  `waitForRequest` calls carry an explicit `{ timeout: 5_000 }` so
  CI worker-contention manifests as a fast failure, not a 30s hang.
- **Approve button locator** uses `/Approve\s+A$/` to match the
  button's accessible name (label + keyshortcut hint `<span>A</span>`)
  and is position-independent — the related "Approve training only" button
  ends in "T" so the regex excludes it cleanly without needing `.first()`.

### 2. New mock state: `MockPhotoApprovalState` in `web/e2e/fixtures/mock-helpers.ts`

Closure-based stateful mock for the four approval dashboard API endpoints:

```
GET  /admin/photos/pending        → { photos: _queue, total_count }
GET  /admin/photos/pending-count  → { count: _queue.length }
PATCH /admin/photos/:id/status    → one-shot override OR splice + success
```

Key details:

- `MockPhotoApprovalItem` interface mirrors `PhotoApprovalItemSchema` from
  `web/src/lib/zod-schemas.ts`. Duplicated rather than imported because
  the `e2e/` directory is outside the `@/*` path alias. Drift is caught
  loudly in-browser by the existing Zod parse in `useAdminPhotoApprovals`
  (malformed mock → loud parse failure, not silent spinner hang).
- `setNextDecideResponse({ status, body })` — one-shot override for the
  next PATCH. Cleared after use. Used by S8 to inject a 409 conflict
  response matching `PhotoApprovalConflictResponseSchema`. Mirrors the
  `setNextUploadResponse` pattern on `MockCollectionPhotoState`.
- Happy-path PATCH handler splices the decided photo out of the queue
  and returns a response matching `PhotoApprovalDecisionResponseSchema`
  (all 8 required fields), derived from the request body + the target
  row. The TanStack Query broad-prefix invalidation on success then
  triggers a refetch, the mock returns the shortened queue, and the
  next photo (or empty state) renders.
- `pending-count` route uses a regex pattern
  (`/\/admin\/photos\/pending-count(\?.*)?$/`) rather than a string glob,
  because Playwright globs do NOT match query strings — code review
  surfaced this as a latent bug (cache-buster query string would fall
  through to the catch-all and Zod would hang).
- Catch-all uses `route.fulfill()`, not `route.continue()`, per the
  `.claude/rules/web-testing.md` rule about catch-alls leaking to the
  real API.

## Technical Details

### Why "pragmatic realistic" over "maximally realistic"

The user asked for "the most realistic for E2E". Phase 2 exploration
surfaced that the existing harness uses `page.route()` mocking for all
domain API data (real auth + mocked data), and that a truly realistic
path (Path A: real HTTP contribution flow) would require:

- Extending `global-setup.ts` with a real-seeding helper
- Precomputing an offline image pair whose dHashes differ by ≤ 4 Hamming
  distance (for the near-duplicate scenario)
- Depending on `PHOTO_STORAGE_PATH` being writable in the test env
- Diverging from every other admin spec in the repo

Given that the API contract is already covered by 905 Vitest integration
tests, and that consistency with `admin-users.spec.ts` + sibling specs
has more long-term value than wire-level fidelity, the pragmatic choice
was confirmed in Phase 3. This checkpoint is still "realistic" in the
sense that matters most: real curator auth, real browser, real TanStack
Router navigation, real TanStack Query invalidation lifecycle, real
`react-hotkeys-hook` document listeners, real Radix dialog portals, and
real Zod parse of mock responses (which catches schema drift).

### Why the `page.addInitScript` localStorage seeding is load-bearing

On first visit, `KeyboardShortcutOverlay` auto-opens a Radix dialog
because `localStorage['photo-approval-shortcuts-seen']` is absent. The
dialog intercepts key events, so decision hotkeys (A / T / R / 1-5)
would be swallowed by the dialog rather than reaching the approval
page. Every E2E test that touches the triage view must seed the
localStorage flag before `goto`, or the first test navigation would
hang with a spurious dialog in the way. Phase 2 research flagged this
as "the single most likely source of spurious 'dialog is in the way'
failures." `page.addInitScript` runs before every page script on every
navigation, so the flag is always in place when the page's own JS
first reads localStorage.

### Why the 409 injection works via a single `_nextDecideResponse` field

`decidePhoto` in `web/src/admin/photos/api.ts` uses raw `apiFetch`
(not `apiFetchJson`) specifically so a 409 response can be parsed with
`PhotoApprovalConflictResponseSchema` and converted into a
`{ conflict: true, current_status, error }` value in the discriminated
union `DecideResult`. This means the mock can prime a real HTTP 409
with a `{ error, current_status }` body and the mutation hook handles
it in the value channel, setting the page's `conflict` state which
renders the amber banner. The test then asserts the banner text and
the Dismiss button's effect. A single one-shot field is enough because
each S8 test only triggers one conflict.

### Spec runtime

8 tests in ~5 seconds on a single worker. This is fast even for mocked
E2E — the speed comes from each scenario doing one `goto` + one
interaction + one assertion with no multi-step navigation and no
toasts to wait for. The design of keeping each scenario tightly scoped
paid off as runtime, not just readability.

## Impact Assessment

- **Reviewer confidence**: the 8 scenarios exercise every distinct code
  path that the reviewer can reach via keyboard and mouse — approve
  public, approve training-only, reject with reason, self-review gate,
  training-only gate, near-duplicate warning, empty state, 409
  conflict. Unit tests cover the negative keyboard-gating cases
  (hotkey suppression during mutation, chord state, etc.).
- **Regression safety**: schema drift between the Zod types in
  `zod-schemas.ts` and the `MockPhotoApprovalItem` interface in
  `mock-helpers.ts` will surface at test run time as a loud parse
  failure, not a silent spinner hang. The mock's `register()` method
  is a single source of truth for all four endpoints, so adding a
  fifth endpoint to the approval dashboard would immediately break
  any test that doesn't update the mock.
- **Maintenance cost**: adding a new scenario (keyboard overlay
  auto-open, prev/next navigation, free-text reject, reject chord) is
  zero new mock infrastructure — one new test using the existing
  factories and `setNextDecideResponse` primitive covers each.
