## Summary

Five fixes and enhancements to the photo approval dashboard surfaced during
manual end-to-end testing of the Phase 1.9b WIP branch. Includes a
privacy-leaking bug (training_only photos appearing in the public catalog),
two UX fixes (approve button semantics, generic toast messages), one
simplification (keyboard chord → plain hotkey), and one new feature
(Hamming-distance near-duplicate warnings in the reviewer sidebar).

## Changes Implemented

### 1. Privacy leak: training_only photos leaked into public catalog (`ceefdf2`)

Five catalog queries filtered by `status = 'approved'` without the
`visibility = 'public'` companion filter. Any admin-approved training_only
photo was surfacing in the public item detail gallery, item list
thumbnails, manufacturer item thumbnails, and search results — directly
contradicting the contributor's `training_only` intent and the policy
documented in `docs/plans/Photo_Contribution_Visibility_Plan.md`.

**Modified:**

- `api/src/catalog/items/queries.ts` — 3 queries fixed:
  - `buildItemsQuery` (item list thumbnail join)
  - `getItemBySlug` base row (item detail header thumbnail)
  - `getItemBySlug` inline photos query (item detail gallery — the big leak)
- `api/src/catalog/manufacturers/queries.ts` — `buildManufacturerItemsQuery` thumbnail join
- `api/src/catalog/search/queries.ts` — `searchCatalog` thumbnail join

**Created:**

- `api/src/catalog/__tests__/visibility-filters.test.ts` — regression guard.
  File-scan test that reads each modified source file and asserts every
  `status = 'approved'` block is colocated with `visibility = 'public'`
  within ±4 lines. Deliberately excludes `ml-export/queries.ts`, which has
  the opposite invariant (training data must include all visibility tiers)
  and its own regression test.

### 2. Admin self-approval allowed (`b77ae0c`)

Relaxed the self-approval guard so admins can decide on their own
contributions. Curators remain blocked. Both enforcement layers updated:
the SQL `can_decide` flag (drives web UI button state) and the PATCH
`/status` runtime guard.

**Modified:**

- `api/src/admin/photos/queries.ts` — `listPendingPhotos` takes new
  `actorRole` parameter; `can_decide` SQL expression gains the
  `$3 = 'admin'` disjunct as a bypass.
- `api/src/admin/photos/routes.ts` — GET `/pending` forwards
  `request.user.role`; PATCH `/:id/status` runtime guard skipped when
  role is admin.
- `api/src/admin/photos/queries.test.ts` — all call sites pass explicit
  `actorRole`; new test asserts `$3 = 'admin'` appears in SQL.
- `api/src/admin/photos/routes.test.ts` — new test: admin decides own
  contribution → 200.

The plain "Approve" button honors contributor intent via the backend's

unqualified "Approve" label reads as "promote to public catalog" and
confused admins during testing. Disable the button instead.

**Modified:**

- `web/src/admin/photos/ActionBar.tsx` — new `canApprovePublic` prop, new

  through to the action bar.
- `web/src/admin/photos/PhotoApprovalPage.tsx` — computes
  `canApprovePublic = activePhoto?.contribution?.intent !== 'training_only'`
  and gates the `A` hotkey so it cannot silently no-op.
- `web/src/admin/photos/__tests__/ActionBar.test.tsx` — new regression test.
- `web/src/admin/photos/__tests__/PhotoTriageView.test.tsx` — default prop updated.

Note: "Approve training only" and "Reject" remain enabled — those are the
only valid decisions for a training_only contribution. Direct curator
uploads (no contribution row) default to public and are unaffected.

### 5. Near-duplicate warning in reviewer sidebar (`8b34307`)

Feature addition during manual testing: surface a warning banner below
the hero image when an existing approved photo for the same item has a
Hamming distance ≤ 4 to the pending photo. The existing-photos sidebar

**Modified:**

  empty dHash get `distance = null` and fall to the end.
- `api/src/admin/photos/schemas.ts` — `existingPhotoItemSchema` adds
  `distance: integer | null` in both `required` and `properties`.
- `api/src/admin/photos/queries.test.ts` — asserts `bit_count` + `distance NULLS LAST`.
- `api/src/admin/photos/routes.test.ts` — fixture updated.
- `web/src/lib/zod-schemas.ts` — `PhotoApprovalExistingPhotoSchema` gains `distance`.
- `web/src/admin/photos/constants.ts` — new `NEAR_DUPLICATE_DISTANCE = 4`.
- `web/src/admin/photos/PhotoTriageView.tsx` — renders warning banner
  below the hero image, above the action bar, when any existing photo
  has `distance ≤ 4`. Banner shows the actual distance for transparency.
- `web/src/admin/photos/__tests__/test-fixtures.ts` — fixture updated.
- `web/src/admin/photos/__tests__/PhotoTriageView.test.tsx` — 3 new tests
  covering the banner present/absent/null branches.

The collection photo contribute mutation's `onError` handler was
swallowing the server's error body and showing a generic
`"Failed to contribute photo"` toast. Users hitting the most common
failure mode — 409 `"This photo has already been contributed"` — had
no visibility into why the action failed.

**Created:**

  `getMutationErrorMessage` helpers, promoted from `admin/lib/` since
  nothing about them is admin-specific.

**Modified:**

- `web/src/collection/photos/CollectionPhotoSheet.tsx` — contribute
  `onError` now passes the caught error through `getMutationErrorMessage`
  so the toast shows the server-provided message.

## Technical Details

### The visibility-filter bug (cause + blast radius)

The ML-export query (`catalog/ml-export/queries.ts`) intentionally
omits a visibility filter because training_only photos must flow into
the training dataset. A regression test there asserts the absence of
any `visibility` filter as a guard. But nothing analogous existed for
the *other* queries, and the inline photo fetch in `getItemBySlug`
appears to have been written fresh without copying the visibility
guard from `catalog/photos/queries.ts` `listPhotos` (which correctly
filters). The new `visibility-filters.test.ts` file-scan test closes
this gap by asserting every `status = 'approved'` block in the three
catalog query files is colocated with `visibility = 'public'`.

### Hamming distance in Postgres

`bit_count(('x' || hex_col)::bit(64))` is the canonical PG14+ pattern
for Hamming distance over hex-encoded hashes. The `'x'` prefix turns
the hex string into a bit-literal (`x'...'` notation) which casts
cleanly to `bit(64)`. `#` is the bit-string XOR operator.

```sql
CASE
  WHEN length(dhash) = 16 AND length(ip.dhash) = 16
  THEN bit_count(('x' || ip.dhash)::bit(64) # ('x' || dhash)::bit(64))::int
  ELSE NULL
END AS distance
```

Per-row cost: two casts + one XOR + one bit_count — all single-
instruction on modern CPUs. The LATERAL subquery runs once per outer
row (≤ 200 pending photos per request), fine for interactive latency.

### The 4-bit threshold

The upload pipeline rejects near-duplicates at Hamming distance ≤ 10,
so any same-item pair surfacing in the approval queue already has
`distance > 10` by construction — the upload guard would have
rejected it otherwise. The review-time threshold is intentionally
tighter (4) because it's catching a different case: (a) the catalog
photo copy going through its own Sharp rescaling, (b) two collection
photos contributed from different uploads that coincidentally
hash-adjacently, or (c) rehash drift. Aligning to 10 would make the
warning silent.

### Backtick-in-template-literal hazard (5b.5 gotcha)

When adding the SQL comment for the Hamming distance block, the
initial version wrapped `bit_count` and similar identifiers in
backticks per normal prose convention. Backticks inside a TypeScript
template literal close the string, so this broke the outer SQL query
into free-floating TS and ESLint's parser caught it as a syntax error
at the stray identifier line. Fix: plain text in SQL comments. Worth
codifying as a rule if it recurs.

## Impact Assessment

- **Security/privacy**: the training_only leak (§1) was a user-visible
  privacy breach — anyone who chose training_only on a contribution was
  having their photo publicly displayed. Fix is immediate on next API
  request; no data migration needed because the filter is query-time,
  not stored-state. No action required from existing contributors, but
  the affected photos continue to feed ML training per the existing
  export pipeline.
- **Reviewer UX**: (§3), (§4), (§5) together tighten the review flow
  based on real curator friction — no ambiguous buttons, no multi-press
  chord, visible duplicate warnings. Should reduce the chance of the
  same photo being approved twice from different collection sources.
- **User feedback loop**: (§6) closes a silent-failure gap on the
  contributor side — 409 "already contributed" is the most common
  failure mode and now surfaces clearly.
- **Admin capability**: (§2) unblocks admins from testing/triage
  scenarios where they need to decide on their own contributions. No
  impact on curators.
